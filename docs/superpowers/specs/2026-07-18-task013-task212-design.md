# Design — Domknięcie TASK-013 (streaming Anthropic) i TASK-212 (spany LLM w OpenAI Agents)

**Data:** 2026-07-18
**Pakiet:** `@tracelyx/core` (`packages/core`)
**Status:** Zatwierdzony do implementacji
**Powiązane taski roadmapy:** TASK-013, TASK-212 (kontrakt: TASK-022, TASK-030, TASK-031, TASK-040)

---

## 1. Kontekst i cel

Dwa SDK-owe taski z Etapów 1–2 są formalnie „done", ale mają realne luki wykryte podczas audytu:

- **TASK-013** — `instrumentAnthropic` patchuje wyłącznie `messages.create` i zakłada odpowiedź non-streaming. Przy `stream: true` nagrywa **mylący, pusty span**: brak tokenów, brak `outputPayload`, a `durationMs` mierzy czas do pierwszego bajtu (span emitowany w chwili dotarcia nagłówków), nie pełną generację.
- **TASK-212** — integracja OpenAI Agents nie emituje **żadnego** spana `llm_call`. `openai.model` leży na spanie `agent_step` jako dekoracyjny atrybut, obok zera tokenów. W efekcie runy OpenAI wnoszą **nic** do agregacji kosztów (TASK-030) i nie dostają niebieskiego paska LLM w waterfall (TASK-022).

**Cel:** domknąć oba taski tak, aby spełniały kryteria akceptacji *realnie* — produkując spany zgodne z kontraktem konsumowanym przez resztę platformy — przy zachowaniu gwarancji **zero-dependency** pakietu.

### Kontrakt spanów (twardy wymóg)

Ustalony podczas researchu (`src/types.ts` + `src/otlp.ts`):

- Agregacja kosztów (TASK-030) grupuje **wyłącznie po top-level polu `llmModel`** (mapowane na `gen_ai.request.model`). Surowy `attributes['openai.model']` / `attributes['llm.model']` jest passthrough i **nie** zasila wymiaru modelu.
- Poprawny span `llm_call` musi mieć na **jednym** spanie: `kind:'llm_call'` + top-level `llmModel` + top-level `promptTokens` + `completionTokens`. `costUsd` liczy backend z modelu + tokenów (SDK go nie ustawia).
- `inputPayload`/`outputPayload` (JSON string) zasilają drawer TASK-022 / viewer TASK-023.

## 2. TASK-013 — instrumentacja streamu Anthropic

### 2.1 Architektura

Jeden punkt instrumentacji pozostaje `client.messages.create` (guard `INSTRUMENTED` bez zmian). `patchedCreate` rozgałęzia się:

```
patchedCreate(params):
  if params.stream === true  → gałąź streamowa (nowa)
  else                       → gałąź non-streaming (obecny kod, bez zmian)
```

`messages.stream()` **nie** jest patchowany osobno — wewnętrznie funneluje przez `create({stream:true})`, więc jedna gałąź obsługuje oba wejścia. Patchowanie obu skutkowałoby **podwójnym liczeniem** jednego wywołania LLM.

### 2.2 Data flow gałęzi streamowej

1. `startTime = Date.now()` przed `await` (jak obecnie).
2. Atrybuty bazowe jak dziś (`llm.model`, `llm.temperature`, `llm.system_prompt_hash`, warunkowo `agent.declared_tools`) + **nowy** `llm.stream = true`.
3. `const stream = await originalCreate(params)`. Jeśli **to** rzuci (auth / 4xx przed rozpoczęciem streamu) → nagraj span `status:'error'` z `error.*` natychmiast i re-throw (brak streamu do owinięcia).
4. Podmiana `stream[Symbol.asyncIterator]` na **delegujący async generator**, który przekazuje każdy event nietknięty konsumentowi i akumuluje stan czystym helperem `accumulateStreamEvent(state, event)`:
   - `message_start` → `state.inputTokens = event.message.usage.input_tokens`; `state.model = event.message.model` (autorytatywny model serwera; fallback `params.model`),
   - `content_block_start` → utwórz `state.blocks[index]` z `event.content_block`,
   - `content_block_delta` → wg `delta.type`: `text_delta`→dopisz tekst; `input_json_delta`→dopisz do bufora `partial_json[index]`; `thinking_delta`/`signature_delta`→analogicznie,
   - `content_block_stop` → sfinalizuj blok (`JSON.parse` zakumulowanego `partial_json` do `tool_use.input`),
   - `message_delta` → `state.outputTokens = event.usage.output_tokens` (kumulatywny; ostatni jest autorytatywny),
   - `message_stop` → `state.sawStop = true`.
5. W bloku `finally` generatora: `finalizeContent(state)` → tablica bloków; `recordSpan` z `SpanPayload`:
   - `kind:'llm_call'`, `name:'anthropic.messages.create'`,
   - top-level `llmModel = state.model`, `promptTokens = state.inputTokens`, `completionTokens = state.outputTokens`,
   - `outputPayload = JSON.stringify(content)`, `inputPayload = JSON.stringify(params.messages)`,
   - `attributes['llm.tool_call_name']` gdy w bloku występuje `tool_use` (mirror logiki non-streaming),
   - `attributes['llm.stream_incomplete'] = true` gdy `!state.sawStop`.
6. `return stream` — **ten sam** obiekt, aby `.controller`, `.tee()`, `.toReadableStream()` działały bez zmian.

### 2.3 Obsługa błędów i edge case (jeden `finally`)

| Sytuacja | Zachowanie |
|---|---|
| Normalne zakończenie | span `status:'ok'`, komplet tokenów/contentu |
| Early break konsumenta | `.return()` generatora → `finally` → span częściowy + `llm.stream_incomplete=true` |
| Błąd w trakcie streamu | re-throw do konsumenta; `status:'error'` + `error.message`/`error.stack`/`error.name`/`error.type` (`classifyError`) |
| Abort (`controller.abort()`) | AbortError przez iterator; `status:'error'`; brak unhandled rejection (jesteśmy w pętli konsumenta) |
| Błąd na `await create()` | span błędu natychmiast, re-throw |

**Timing:** span nagrywany na **zakończeniu** streamu (`finally` generatora), nie przy resolve `create()` — inaczej `durationMs` obejmowałby tylko czas do pierwszego bajtu.

### 2.4 Znany, świadomy gap

Konsument używający **wyłącznie** `stream.tee()` lub `stream.toReadableStream()` omija `[Symbol.asyncIterator]` (te metody wołają prywatny `iterator()` bezpośrednio) → jego stream pozostaje nieotraceowany. Rzadki wzorzec; udokumentowany w README, poza zakresem tej iteracji.

### 2.5 Testy (mocki strukturalne, zero SDK)

Helper `makeStreamMock(events)` — obiekt, którego `[Symbol.asyncIterator]` to async generator zwracający fabrykowane surowe eventy; opcjonalnie stuby `controller`/`tee`.

Przypadki:
1. Happy path — jeden span `llm_call`; `promptTokens`/`completionTokens` z eventów; `outputPayload` = zrekonstruowany JSON; `llm.stream=true`; `status:'ok'`; `durationMs` obejmuje całą pętlę.
2. Early break — po kilku eventach `break` → span częściowy + `llm.stream_incomplete=true`.
3. Błąd w trakcie — generator rzuca po kilku eventach → pętla odrzuca, `status:'error'`.
4. `tool_use` w streamie → `llm.tool_call_name` ustawiony.
5. Błąd na `await create()` → span błędu, brak streamu.
6. Ścieżka `messages.stream()` (fake `MessageStream` wołający patchowany `create`) → **dokładnie jeden** span (brak double-count).

Istniejące testy non-streaming pozostają zielone.

## 3. TASK-212 — spany `llm_call` przez wstrzyknięty tracing processor

Wybrane podejście: **hybryda (Opcja C)** — zachowujemy istniejące `agent_step`/`tool_call` bez zmian (w tym `openai.model` na `agent_step` jako model „zadeklarowany") **oraz** dodajemy prawdziwe spany `llm_call` przez tracing processor SDK.

### 3.1 API (opt-in, zero-dependency przez injection)

```ts
instrumentOpenAIAgents(agentOrRunner, tracelyxClient, options?: {
  tracing?: AddTraceProcessorFn   // np. addTraceProcessor z @openai/agents
})
// gdy options.tracing podane → options.tracing(new TracelyxTracingProcessor(tracelyxClient))
```

Rdzeń **nigdy** nie importuje `@openai/agents` (pakiet nie jest zadeklarowaną zależnością — jest instalką hoistowaną). `addTraceProcessor` podaje wywołujący (ma SDK w swojej aplikacji). Brak `tracing` → zachowanie identyczne jak dziś (żadnych spanów `llm_call`). Rejestracja chroniona przed podwójnym wywołaniem.

### 3.2 `TracelyxTracingProcessor`

Klasa implementująca **strukturalnie** (bez importu typów SDK) minimalny interfejs `TracingProcessor` (`onSpanEnd` wymagane; `onTraceStart/onTraceEnd/onSpanStart/shutdown/forceFlush` no-op/opcjonalne).

`onSpanEnd(span)`:
- `span.spanData.type === 'response'` (domyślna ścieżka Responses API): `model = spanData._response.model`; tokeny z `spanData._response.usage.{input_tokens, output_tokens}`; `response_id = spanData.response_id`.
- `span.spanData.type === 'generation'` (Chat Completions): `model = spanData.model`; tokeny z `spanData.usage` **lub** `spanData.output[0].usage` — ⚠️ **do weryfikacji runtime** (patrz §3.4).
- inne typy spanów SDK → ignoruj.

Emitowany `SpanPayload`:
- `kind:'llm_call'`, `name:'openai.response'` / `'openai.generation'`,
- top-level `llmModel`, `promptTokens`, `completionTokens`,
- `attributes['openai.model'] = model` (dla spójności/diagnostyki),
- `startTime`/`endTime` z `Date.parse(span.startedAt / span.endedAt)` (ISO stringi),
- `status` z `span.error` (`'error'` gdy obecny, inaczej `'ok'`).

### 3.3 Linkage — cel i mechanizm

**Cel:** span `llm_call` w **tym samym trace** i zagnieżdżony pod właściwym `agent_step`.

- **Primary — bridge przez `AsyncLocalStorage`.** Wywołanie modelu odbywa się wewnątrz `runWithContext({ spanId: agentStepId, traceId }, () => originalRun(...))` z patchowanego `run()`. Jeśli SDK woła `onSpanEnd` synchronicznie w tym kontekście, `getActiveContext()` w procesorze zwróci kontekst `agent_step` → ustawiamy `traceId` = kontekstowy, `parentSpanId = agentStepId`. Zero bookkeepingu.
- **Fallback** (gdy `getActiveContext()` puste, bo `onSpanEnd` odpalił poza kontekstem runu): mapa korelacji `Map<sdkTraceId, { traceId, agentStepSpanId }>` zakładana w patchowanym `run()`. Klucz = trace id SDK odczytany na starcie runu.
- **Ostateczny fallback:** span jako top-level orphan — TASK-022 degraduje łagodnie (top-level + ikona ostrzeżenia).

### 3.4 Ryzyko wymagające weryfikacji runtime

Dwie rzeczy potwierdzić na realnym `@openai/agents@0.13.0` **przed finalizacją**:
1. Czy `onSpanEnd` dla spana modelu odpala się **synchronicznie w ALS-kontekście** patchowanego `run()` (decyduje, czy primary linkage działa, czy schodzimy na fallback z mapą).
2. Dokładne pole tokenów na spanie `'generation'` (`spanData.usage` vs `spanData.output[0].usage`).

Ścieżka `'response'`/Responses API (domyślna) jest potwierdzona researchem.

### 3.5 Testy (bez realnego SDK)

1. `onSpanEnd` testowany bezpośrednio fabrykowanymi literałami span:
   - `response`: `{ spanData:{ type:'response', response_id, _response:{ model:'gpt-4o', usage:{ input_tokens:11, output_tokens:22 } } }, traceId, spanId, parentId, startedAt, endedAt, error:null }` → asercja `llm_call` z `openai.model='gpt-4o'`, `promptTokens=11`, `completionTokens=22`.
   - `generation`: analogicznie dla ścieżki Chat Completions.
2. Injection — `instrumentOpenAIAgents(agent, client, { tracing: fakeAddTraceProcessor })` rejestruje procesor (fake przechwytuje instancję).
3. Bridge ALS — `onSpanEnd` wywołany wewnątrz `runWithContext(...)` → `parentSpanId` = aktywny span; poza kontekstem → fallback/orphan.
4. Guard podwójnej rejestracji.
5. Istniejące testy `agent_step`/`tool_call` (w tym asercje `openai.model` na `agent_step`, linie ~40 i ~376) pozostają zielone.

## 4. Zakres wspólny i non-goals

**W zakresie:**
- Aktualizacja `packages/core/README.md`: nota o instrumentacji streamu (TASK-013) i o opcji `{ tracing }` (TASK-212).
- Zero nowych `dependencies` w `package.json` (`@openai/agents` co najwyżej type-only/injected).

**Non-goals (świadomie poza zakresem):**
- Kalkulacja `costUsd` w SDK — liczy backend z modelu + tokenów.
- Atrybuty security `llm.tool_call_name` / `agent.declared_tools` na `llm_call` OpenAI — to TASK-040.
- Wsparcie przy **wyłączonym** tracingu SDK OpenAI — byłaby to Opcja B2 (wrapper Model/Provider), świadomie odrzucona jako inwazyjna.
- Zmiana zachowania spanów `agent_step` / `tool_call`.
- Traceowanie konsumentów Anthropic używających wyłącznie `tee()`/`toReadableStream()`.

## 5. Kryteria akceptacji (Definition of Done)

- [ ] Early break / błąd / abort w streamie obsłużone bez rzucania do kodu użytkownika ponad oryginalny błąd; `llm.stream_incomplete` przy niepełnym streamie.

- [ ] Bez `{ tracing }` zachowanie i istniejące testy niezmienione.
- [ ] Runtime-probe (§3.4) wykonany i wynik odzwierciedlony w kodzie.
- [ ] Zero nowych runtime dependencies; `pnpm test` i `pnpm check-types` zielone.
